# 独立战斗场景系统设计

> 状态：**已批准，进入 P0 实施**（Q1～Q8 于 2026-07-23 全部按推荐方案批准）
> 范围：S02 地图 + S05 军事 + S10 战斗，预留 S16 存档边界
> 本文是独立战场系统的正式架构与 P0～P6 分期依据；各期仍须按路线图逐项实现、验证与记录。
> 编号说明：现有文档已使用 00～20，故本文使用 21；本系统是既有 S02/S05/S10 的核心深化，**不新增第 23 个并列大系统**。

---

## 一、目标与非目标

### 1.1 目标

出征不再直接在行政大地图上完成整场战斗。军队进入敌对目标郡国后，创建一份与该场战争绑定的独立战场场景：

```text
行政大地图
  → 出征与跨郡行军
  → 进入目标郡边界 / 发生敌对接触
  → 独立郡域战场（历史郡治、属县、道路、水系、关渡）
  → 两军在同一战场节点接触
  → 自动结算 / 标准模式 / 六角微操
  → 结果回写独立战场
  → 战争结束后原子回写行政大地图
```

同一目标郡在相同历史切片下具有稳定的县城、方位和交通骨架；随机性只能影响部署、天气、遭遇等动态战况，不能把南郡随机生成成另一套行政地理。

### 1.2 非目标

- 本轮不写引擎、UI、JSON、Zod 或迁移代码。
- 不在原型期录入全部东汉郡县。
- 不追求现代测绘级精确边界；“大致方位”必须可追溯并带置信度。
- 不复制第三方专有地图瓦片、截图或未经授权的现代历史地图。
- 不另建随机源，不允许 `Math.random()` 参与场景生成或战斗结算。
- 不把县级战场节点立即升级为行政大地图上的完整经济实体。

## 二、现状审计

### 2.1 当前 30 城是否已有“郡”层级

**【现状事实】有郡国语义，没有独立郡国层级。**

`server/src/data/cities.json` 当前实有 30 条记录。每条记录本质上是一个郡国治所节点：

- `name`：画面显示的治所名，如“江陵”；
- `adminName?`：正式郡国名，如“南郡”；
- `province`：州，如“荆州”；
- `x/y`：行政大地图像素坐标；
- `countyCount?`：下辖县数的摘要值；
- 大地图道路由 `city-roads.ts` 等邻接关系表达。

目前没有：

- 独立 `Commandery` 实体；
- 属县的名称、史料时期、县级 ID；
- 县城的经纬度或郡内相对坐标；
- 郡内道路、水系、渡口、山口；
- 治所迁移、郡县改隶等按剧本生效的历史版本。

因此，现有数据只能回答“江陵节点代表南郡”，不能生成“南郡内江陵、当阳、华容、枝江等节点之间的历史战场”。

### 2.1.1 大地图节点与史实郡县分离原则

**【已批准方案】行政大地图战略节点与郡域历史属县是两套独立、允许不完全重合的模型。**
史实上从属于某郡的城市，可因其跨郡交通、军事或政治可玩性而继续作为独立大地图节点；此时
郡域模板不得再生成同名可争夺县城，只能以边界入口或邻郡方向标签引用该战略节点。190 年襄阳
史实属南郡、游戏中保留为独立战略节点，是首个适用案例；这种偏离必须逐例留有史实依据与
可玩性说明，不能反向篡改史实建制，也不能让同一地点出现两个权威控制实体。

### 2.2 当前战场架构

**【现状事实】独立战场壳已存在，但地理来源仍是行政节点子集。**

仓库已有：

- `BattlefieldMap`：Tier I 战场地图，已进入 `GameState`；
- `MeleeState`：Tier II 标准白刃战；
- `BattleState`：六角微操战斗；
- `extractBattlefieldNodes()`：以目标城、出发城及其邻接点抽取战场；
- Zustand 的 `battlefield` / `melee` 状态和对应画面；
- 进行中三级战斗的完整快照 Schema；
- 可序列化 `xorshift32-v1` 权威随机流。

本设计不是从零创建“独立战场”，而是把 Tier I 的数据源从“30 城大地图邻接切片”替换为“目标郡的历史地理模板 + 本次战争实例”，并理顺其与六角层的职责。

### 2.3 现有文档规模口径

**【现状事实】**

- 0-A 验收基线：30 个治所节点。
- `08-data-dictionary.md` 规划全量：105 个郡国记录。
- 现有 `countyCount` 只是摘要且当前数据多为 3～7 的演示值，不能当作史实属县表。

本设计涉及未来静态数据规模时，仍以 `08-data-dictionary.md` 为数字真源；本文只提出分期建议，不直接改变规模数字。

## 三、总体架构

### 3.1 推荐分层

**【已批准方案】采用“历史郡域节点层 + 可选六角战术层”，不二选一。**

| 层级 | 空间含义 | 主要内容 | 是否必经 |
|------|----------|----------|:--------:|
| 行政大地图 | 州郡战略空间 | 出征、跨郡路线、国境、全局补给 | 是 |
| 独立郡域战场 | 一郡之内的战役空间 | 郡治、属县、道路、水系、关渡、Army 行军与争夺 | 是 |
| 标准白刃战 | 一次接战的回合决策 | 阵型、战术点、计谋、单挑 | 玩家选择 |
| 六角微操 | 一次接战的局部战术空间 | 单位移动、地形、攻击、战法 | 玩家选择 |
| 单挑演出 | 局部人物对决 | 既有 duel 引擎与演出 | 条件触发 |

“独立战场”首先指郡域战役场景，不等于把整个郡铺成一个超大六角棋盘。县城之间可能相距数十至数百里，若用同一六角比例尺承载，移动、射程和城池尺寸会失真。

六角格继续存在，但只在两军于同一县城、道路、渡口等地点接战时，生成或载入该地点的局部战术地图。这样既保留现有 S10 投资，又让历史郡县真正参与战争过程。

### 3.2 静态快照与动态实例

独立战场不是纯静态图片，也不是全随机地图，而是两层组合：

1. **历史地理模板（静态、版本化）**
   - 郡治、属县、史实交通与地貌锚点；
   - 节点的相对位置和稳定 ID；
   - 对应历史有效期与资料出处。
2. **战争场景实例（动态、可存档）**
   - 参战 Army、控制权、驻军与城防；
   - 已侦察区域、营寨、陷阱、桥梁损毁；
   - 天气、部署、遭遇、战场回合；
   - 模板版本与生成记录。

同一模板可以服务多个剧本和多场战争；动态结果不得反写静态史料数据。

## 四、地理数据模型

### 4.1 标识与层级

**【已批准方案】新增独立静态目录，不把所有县塞入 `cities.json`。**

```typescript
type HistoricalConfidence = 'attested' | 'approximate' | 'inferred';

interface CommanderyDefinition {
  id: string;                     // 稳定 ID，如 jing_nanjun
  name: string;                   // 南郡
  province: string;               // 荆州
  seatCountyId: string;           // 郡治县 ID
  worldCityId: number;            // 关联当前行政大地图治所
  validFromYear?: number;
  validToYear?: number;
  variantOf?: string;             // 改置/迁治时关联的基础定义
  countyIds: string[];
  localBounds: LocalBounds;
  sourceRefs: string[];
}

interface CountyDefinition {
  id: string;                     // 稳定且不依赖显示名
  name: string;
  commanderyId: string;
  role: 'seat' | 'county' | 'marquisate' | 'frontier';
  validFromYear?: number;
  validToYear?: number;

  // 历史地理层
  lon?: number;                   // 有可靠公开依据时填写
  lat?: number;
  localX: number;                 // 0..1 郡域归一化坐标，渲染真源
  localY: number;                 // 0..1
  confidence: HistoricalConfidence;
  locationNote?: string;
  sourceRefs: string[];

  // 战场模板层
  terrainTags: Array<
    'plain' | 'hill' | 'mountain' | 'forest' |
    'river' | 'lake' | 'marsh' | 'coast'
  >;
  adjacentCountyIds: string[];
  landmarkIds: string[];
}

interface HistoricalRouteDefinition {
  id: string;
  commanderyId: string;
  fromNodeId: string;
  toNodeId: string;
  kind: 'road' | 'river' | 'pass' | 'ferry';
  movementCost: number;
  seasonal?: 'all' | 'dry' | 'wet';
  confidence: HistoricalConfidence;
  sourceRefs: string[];
}

interface BattlefieldLandmarkDefinition {
  id: string;
  commanderyId: string;
  name: string;
  kind: 'river' | 'lake' | 'marsh' | 'mountain' |
        'pass' | 'ferry' | 'bridge' | 'port';
  localGeometry: unknown;         // 实装时收紧为 point/polyline/polygon 判别联合
  tacticalTags: string[];
  confidence: HistoricalConfidence;
  sourceRefs: string[];
}
```

### 4.2 为什么同时保留经纬度与相对坐标

- `lon/lat` 用于与行政大地图、公开地理底图和未来校正工具对齐。
- `localX/localY` 是独立战场稳定布局的渲染真源，不受底图投影和画布尺寸影响。
- 史书常能确认县名和相对关系，却不能提供现代精确坐标；这类节点可只填归一化位置并标 `approximate` / `inferred`。
- 不允许用随机数填补未知县址。未知位置由人工编订的近似布局表达，并在数据中保留置信度和说明。

### 4.3 行政实体与战场节点分离

县城应先作为**历史地理/战场模板实体**，不自动成为完整 `City`：

- `City`：全局行政经济实体，有人口、金粮、武将、政策等完整状态。
- `CountyDefinition`：静态历史地理记录。
- `BattlefieldNodeInstance`：某场战争中由县定义派生的可争夺节点。

这样原型无需一次性解决数百或上千县的经济模拟、AI、迷雾和存档膨胀。未来若 0-B 决定县城进入行政模拟，再通过稳定 `countyId` 升格，不破坏战场存档引用。

### 4.4 剧本时间与行政沿革

郡县设置随年代变化，不能只维护一张“东汉通用表”。生成器按以下顺序解析：

1. 读取剧本年月；
2. 选择该年有效的 `CommanderyDefinition` / `CountyDefinition`；
3. 应用剧本专属覆写（迁治、析置、临时控制）；
4. 冻结 `templateVersion` 和实际节点清单到战争场景；
5. 战争进行中跨年时不热替换模板，避免县城凭空出现或消失。

### 4.5 史料与授权

**【已批准方案】史名、隶属、治所优先级如下：**

1. 《后汉书·郡国志》/《续汉书·郡国志》等公有领域原典；
2. 《三国志》及裴注中明确的建安以后改置；
3. 《水经注》等公有领域原典用于水系与相对方位；
4. 现代研究仅作校核；若要复制其整理成果或图件，必须单独核验许可。

每条数据使用 `sourceRefs` 指向统一来源目录，至少记录书名、卷/志、条目、版本或公开文本 URL、访问日期。原典本身属公有领域，但网站扫描图、现代标点整理和现代地图不当然可再分发。

首批可登记的公开原典入口：

- [中国哲学书电子化计划《后汉书》](https://ctext.org/hou-han-shu/zh)（含《郡国志》分卷与图文对照入口）；
- [维基文库《后汉书》卷一百一十二](https://zh.wikisource.org/zh/%E5%BE%8C%E6%BC%A2%E6%9B%B8/%E5%8D%B7112)（含南郡条，适合作为人工复核入口）。

链接只用于定位公有领域原典，不代表可直接复制网站整理数据或扫描资产；正式数据仍应记录所据底本并交叉校勘。

正式录入时同步更新 `CREDITS.md` 的“古籍/历史文献引用”和“地理坐标”，处理方式与现有 Natural Earth / 古籍声明一致。原型数据提交前应做人工校勘，不把网络百科当唯一真源。

### 4.6 原型郡选择

**【推荐】首个原型选南郡，第二个选颍川郡或汉中郡。**

- 南郡：正好覆盖用户提出的江陵；江河、湖沼、渡口与陆路并存，可验证水陆地貌。
- 颍川郡：偏陆战和平原道路，可验证系统并非只为荆州水网特制。
- 汉中郡：山谷、关隘明确，但首期即选会提高地形与路径复杂度。

南郡县名原型可从《后汉书·郡国志》所载条目建立候选清单；具体采用哪个年代切片、哪些县进入首期，仍须校勘后定案。本文不把未经逐条校勘的县表写成数据真源。

## 五、战场场景构成

### 5.1 郡治与属县的角色

**【已批准方案】县城是可占领、可驻守、可围攻的战役目标，不只是背景标签。**

| 节点 | 战役作用 | 战术作用 |
|------|----------|----------|
| 郡治 | 战争主目标；最高城防、补给与政治权重 | 可触发攻城六角模板/标准模式 |
| 属县 | 补给、驻军、侧翼路线、阶段性控制 | 可触发县城郊野/守城接战 |
| 关隘 | 封锁路线、提高防守效率 | 山口/关城局部地图 |
| 渡口/桥梁 | 控制跨水通路，受季节和破坏影响 | 渡河/桥头局部地图 |
| 港口 | 水军进出与补给 | 水陆接驳局部地图 |
| 山林湖泽 | 影响可通行边和遭遇修正 | 作为局部战术地形来源 |

原型期县城控制先只影响本场战争，不立即改写全局县级经济。战争结束后再按结算规则汇总为郡国控制结果。

### 5.2 战场边界

**【已批准方案】默认边界为“目标郡全境”，不是目标城周边随机窗口，也不是无限扩张到邻郡。**

战场包含：

- 目标郡治和该历史切片中纳入模板的全部属县；
- 郡内关隘、渡口、港口和关键地貌；
- 每条合法入侵路线对应的**边界入口节点**；
- 必要时显示邻郡方向标签，但不加载邻郡全部县城。

跨入邻郡时结束当前郡域场景并创建/载入另一场战争场景。这样边界稳定、存档体积可控，也让“夺取一郡”的军事目标清晰。

例外：边界争夺、跨郡围城或历史事件可用 `scenarioOverrides` 引入少量郡外前沿节点，但必须显式列举，不由生成器任意扩张。

### 5.3 历史地图与可玩性

场景生成采用“史实骨架 + 可玩性标注”：

- 县名、郡属、治所和大致方位是静态事实层；
- 道路与水路尽量有史据，无法确认时标 `inferred`；
- 节点间距可为画面和操作适当变形，但保持方位、上下游和邻接关系；
- 战略点可在既定路线中配置多个候选槽位；
- 随机数只能选择候选槽位或动态状态，不能改变史实县城的位置和隶属。

### 5.4 六角战术地图来源

若玩家选择微操，局部六角图由当前接战节点派生：

```typescript
interface TacticalMapRequest {
  battlefieldId: string;
  encounterId: string;
  nodeId: string;
  nodeKind: 'seat' | 'county' | 'pass' | 'ferry' | 'road' | 'field';
  terrainTags: string[];
  landmarkRefs: string[];
  attackerApproach: string;
  defenderPreparation: string;
}
```

原型期使用少量固定模板（江陵城郊、县城、江岸渡口等），按历史节点标签选模板；后期才增加确定性程序生成。六角坐标只表达局部战术距离，不与郡域 `localX/localY` 直接换算。

## 六、场景切换与状态流

### 6.1 触发流程

**【已批准方案】创建战场与真正接战分为两个时点。**

1. 玩家在行政大地图选择己方相邻节点可达的敌方郡治或合法边界目标；
2. 服务端校验编成、粮草、外交状态和跨郡路径；
3. 创建 `War` / `BattlefieldInstance`，记录目标郡、进攻入口和参战 Army；
4. Army 在行政层走到目标郡边界后，切换到独立郡域战场；
5. Army 从对应边界入口进入，沿郡内道路/水路行军；
6. 敌对 Army 同节点、攻击驻军或强攻城池时，创建 `Encounter`；
7. 玩家选择自动、标准或六角微操；
8. 接战结果回写郡域战场，战争未结束则返回该战场；
9. 达成胜负/撤军条件后，统一结算并回写行政大地图。

### 6.2 邻接与接触判定

- **能否出征**：由行政大地图的跨郡邻接和道路网络决定。
- **从哪里进入战场**：由跨郡道路映射到目标郡的边界入口。
- **是否发生战斗**：由郡域战场内 Army 与敌军/敌方节点接触决定。
- **攻城条件**：Army 抵达郡治或属县节点，且守军/城防未被绕过。

因此，“大地图相邻”只授权进入目标郡，不再等价于立刻扣兵或占城。

### 6.3 战争结束条件

原型建议：

- 攻方占领郡治并击退仍能立即反攻的主力，判进攻胜利；
- 攻方主力全灭、投降或撤回边界，判防守胜利；
- 回合上限、补给耗尽或双方脱离，可判僵持；
- 属县控制影响补给、结算评价和未来郡国归附，但原型不要求攻方逐县占领后才能结束。

完整版再接入现有 §17.6 郡国归属算法，使郡治失守后未占属县可能投降、中立或继续抵抗。

### 6.4 回写事务

战争结算必须通过单一服务端事务函数完成，禁止前端分别修改城、Army 和武将：

```typescript
interface BattlefieldSettlement {
  battlefieldId: string;
  outcome: 'attacker_victory' | 'defender_victory' | 'stalemate' | 'withdrawal';
  cityControlChanges: Array<{ cityId: number; from: number | null; to: number | null }>;
  countyControlSummary: Array<{ countyId: string; ruler: number | null }>;
  armyUpdates: Array<{
    armyId: string;
    troops: number;
    morale: number;
    food: number;
    destinationNodeId: number;
    phase: string;
  }>;
  officerUpdates: Array<{
    officerId: number;
    status: 'safe' | 'wounded' | 'captured' | 'killed';
  }>;
  spoils: { gold: number; food: number };
  auditLog: string[];
}
```

事务不变量：

- 一支 Army 不能同时属于两个活动战场；
- 伤亡不能在 Encounter 与战争结算阶段重复扣除；
- 城池控制、势力城市列表、驻军、武将位置必须一致；
- 撤退必须落到可达且合法的己方/中立入口；
- 同一 `settlementId` 重放必须幂等。

### 6.5 前端场景管理

**【已批准方案】不新增浏览器路由；沿用 Zustand，但升级为明确的场景栈。**

理由：

- 这些画面共享同一局游戏和权威服务端状态，不是可独立深链的页面；
- 当前已有 `screen`、`battlefield`、`melee`，复用成本低；
- `docs/02-architecture.md` / `07-ui-design.md` 已预留屏幕栈技术债 D-0B-6。

建议收敛命名：

```typescript
type Scene =
  | 'scenario'
  | 'world'
  | 'battlefield'
  | 'melee'
  | 'tactical'
  | 'duel';

interface SceneFrame {
  scene: Scene;
  battlefieldId?: string;
  encounterId?: string;
}
```

`boot`/加载错误属于应用壳状态，不进入游戏场景栈。服务端 `GameState` 仍是权威；场景栈、镜头、选中节点、动画进度是前端瞬态状态，不进入存档。

## 七、与现有系统的集成边界

### 7.1 六角战斗引擎

**【已批准方案】迁移适配，不推倒重做。**

保留：

- 六角坐标、移动与攻击；
- 兵种克制、阵型、地形、天气；
- 火计、战法、暴击/反击/连击；
- 权威 RNG 注入；
- 六角单挑入口。

新增适配层：

```text
Encounter + 历史节点标签
  → TacticalMapRequest
  → 固定/确定性六角模板
  → CampaignArmy/Squad 映射 BattleUnit
  → 既有 BattleState 引擎
  → TacticalBattleResult
  → 回写 Encounter / BattlefieldInstance
```

需要废弃的是“行政大地图相邻即直接创建抽象六角战斗”的入口假设，不是六角规则本身。

### 7.2 标准模式与自动结算

三种入口必须消费相同的战前快照并输出统一 `EncounterResult`：

- 自动结算：现有 `runAutoBattle` 的演进版；
- 标准模式：现有 `runMeleeRound`；
- 微操：现有 `BattleState`。

长期应共享伤亡、俘虏、武将状态和战后回写规则，避免不同入口出现“同兵力不同数据语义”。

### 7.3 单挑

**【已批准方案】单挑保留两种触发语境，共用同一 duel 引擎。**

1. **阵前/城下挑战**：两军同节点但尚未全面接战时，由 Tier I 战场动作发起；影响士气、接战先手或守军状态。
2. **战中单挑**：标准模式或六角微操中由既有条件触发。

两者只在触发前置、拒绝代价和结果回写上不同，不复制伤害、指令、受伤和 RNG 规则。原型期可只保留战中单挑，城下挑战后置。

### 7.4 S02、S05、S10 职责

| 系统 | 负责 | 不负责 |
|------|------|--------|
| S02 地图 | 历史地理模板、坐标投影、道路水系显示 | 战斗伤害与胜负 |
| S05 军事 | 出征、Army、跨郡入口、补给、撤退、战争结算 | 六角行动细节 |
| S10 战斗 | Encounter、标准/微操结算、单挑、战术结果 | 郡县史料真源 |
| S16 存档 | 模板版本、实例状态、RNG 状态、迁移 | 运行时动画与镜头 |

## 八、存档与序列化预留

### 8.1 本次设计是否预留

**【已批准方案】必须现在预留，原型即可中途存档。**

S16 当前虽没有生产存取 UI/SQLite，但 `GameState` 完整 Schema 已覆盖进行中 `BattlefieldMap`、`MeleeState`、`BattleState` 与 PRNG 状态。新系统不能倒退为“战斗中禁止存档”。

### 8.2 建议实例结构

```typescript
interface BattlefieldInstance {
  id: string;
  warId: string;
  templateId: string;
  templateVersion: number;
  scenarioDateAtCreation: string;
  targetCommanderyId: string;
  entryNodeIds: string[];

  nodeStates: BattlefieldNodeState[];
  routeStates: BattlefieldRouteState[];
  armyIds: string[];
  encounters: EncounterState[];
  turn: number;
  phase: 'active' | 'settling' | 'resolved';

  generationAudit: {
    rngAlgorithm: 'xorshift32-v1';
    rngDrawStart: number;
    rngDrawEnd: number;
    decisions: string[];
  };
}
```

### 8.3 保存什么、不保存什么

保存：

- 模板 ID/版本和创建时的实际节点引用；
- 节点控制、城防、驻军、设施、陷阱、道路状态；
- Army 位置、路径、补给、伤亡；
- Encounter、标准模式、六角战斗和单挑的进行中状态；
- 权威 PRNG 的寄存器与消费计数；
- 生成审计记录。

不保存：

- Konva 图层、粒子、补间动画；
- 当前 hover、镜头缩放、临时选择框；
- 可从模板和实例状态纯函数重建的绘制缓存。

### 8.4 模板版本迁移

存档加载不能静默套用最新版县表。规则：

1. 进行中战场优先读取存档冻结的 `templateVersion`；
2. 版本仍受支持时，用旧模板重建；
3. 必须迁移时走显式迁移器，并记录县 ID 映射；
4. 找不到旧模板且无迁移器时拒绝加载，不猜测节点；
5. 已结算战争无需保留完整模板，只保留审计摘要。

## 九、RNG 与确定性

### 9.1 接口原则

**【确定约束】所有随机函数必须显式注入权威随机源。**

```typescript
interface RandomSource {
  next(): number; // [0, 1)
}

interface BattlefieldGenerationContext {
  random: RandomSource;           // 服务端权威 xorshift32-v1
  template: CommanderyTemplate;
  attacker: CampaignArmy;
  defender: CampaignArmy[];
  scenarioDate: string;
}
```

生产服务层传入现有 `SerializableRng` / `runtimeRandom` 的适配器；纯函数不得提供 `Math.random()` 默认值。测试使用固定种子的同接口实例。

### 9.2 静态与随机边界

不消费 RNG：

- 郡治与属县清单；
- 县城相对位置；
- 史实道路、水系、关隘；
- 战场边界与入侵入口映射；
- 模板选择（若条件唯一）。

允许消费 RNG：

| 随机点 | 用途 | 原型期 |
|--------|------|:------:|
| 防守军初始部署 | 在合法候选节点中选择驻防 | 可选 |
| 同节点内 Squad 站位 | 六角初始部署 | 是 |
| 开战天气 | 从地区/季节允许集合抽取 | 可选 |
| 天气变化 | 战斗中动态天气 | 沿用现有 |
| 遭遇事件顺序 | 斥候、伏击、迷路等候选事件 | 后置 |
| 伏击/陷阱发现 | 成功率判定 | 后置 |
| AI 战场行动 | 目标/路径/战术选择 | 后置，若要求整场复现则必须收口 |
| 伤害浮动 | 既有六角/标准/自动结算 | 已有 |
| 暴击/反击/连击 | 既有战斗事件链 | 已有 |
| 单挑指令及判定 | 既有 duel 流 | 已有 |
| 俘虏/受伤/战利品 | 战后结算 | 已有或待统一 |
| 可玩性战略点槽位 | 固定候选中的启用/类型选择 | 后置 |

### 9.3 消费顺序与审计

- 每个入口的消费顺序写入契约测试，不能依赖对象键遍历或客户端帧率。
- 遍历 Army、节点、事件候选前先按稳定 ID 排序。
- 生成完成记录 `rngDrawStart/rngDrawEnd`；调试日志记录决策标签，不必记录每个浮点值。
- “无随机配置”必须零消费，防止新增装饰逻辑改变后续战斗结果。
- 读档后从信封中的 `state + draws` 继续，不由 `battlefieldId` 私自派生第二 RNG。
- 若未来需要互不干扰的子流，必须先升级权威 PRNG 协议和存档版本；本设计不预设哈希分叉。

### 9.4 确定性验收

同一模板、输入状态和 RNG 快照：

1. 生成的动态部署与审计记录完全一致；
2. 保存后恢复再继续，结果与不中断运行一致；
3. UI 重绘、缩放、切换画面不消费 RNG；
4. 静态模板生成不调用 RNG；
5. 全仓扫描确认新增路径无 `Math.random()`。

## 十、0-A 与 0-B 数据规模策略

### 10.1 原则

Phase 0-A 的目的仍是验证架构，不把“30 个治所”误解为“必须一次补齐这 30 郡的全部属县”。县级数据采用纵向切片：

| 阶段 | 郡域数据 | 用途 |

| 原型 P1 | 2 郡对照切片 | 验证水网与陆地两种地理 |

| 0-B | 105 郡国逐批补齐 | 全量历史地理覆盖 |

- 有模板的郡：进入新独立历史郡域战场；

已批准采用 Q5-A：开发构建允许明确标注的旧简化兼容路径，正式内容只开放已有历史模板的战争。

## 十二、风险与护栏

| 风险 | 后果 | 护栏 |
|------|------|------|
| 把郡域做成巨型六角图 | 比例尺失真、操作冗长 | 郡域节点层与局部六角层分离 |
| 先录全量再验架构 | 数据返工巨大 | 1 郡纵向切片先行 |
| 县名有史据但位置无史据 | 伪精确 | `confidence` + `locationNote`，不随机补坐标 |
| 不同年代混用郡县 | 历史错误 | 有效期 + 剧本覆写 + 模板冻结 |
| 县城直接升级完整 City | AI/经济/存档范围爆炸 | 静态县定义与行政实体分离 |
| 三种战斗入口各自结算 | 重复扣兵、状态漂移 | 统一 `EncounterResult` 和回写事务 |
| 新生成器私建 seed | 读档不确定 | 只注入权威 `xorshift32-v1` |
| UI 状态写入权威存档 | 存档膨胀、兼容困难 | 只存玩法状态，不存镜头动画 |
| 现代地图/扫描授权不明 | 版权风险 | 原典文本优先，逐资产登记许可 |

## 十三、已批准决策（Q1～Q8）

以下决定已于 2026-07-23 全部按推荐方案批准，构成本系统实施基线。

### Q1（最高影响）：是否保留六角格战术层（已批准：2026-07-23）

- **A．推荐：保留为可选微操层。** 郡域历史节点是必经战场；接战时可选自动、标准或六角微操。复用现有 S10 最大，工作量中等。
- B．六角成为每次接战必经层。战术最细，但节奏慢、UI 和 AI 工作量最大。
- C．彻底取消六角，只保留郡域节点 + 标准/自动。架构更轻，但会废弃大量现有能力。

本文按 A 展开。

## 十五、Q1～Q8 已批准决策摘要（按影响范围从大到小）

> 本节完整保留 §十三各问题及选项的原意、专业推荐与影响判断；推荐项已于 2026-07-23 全部批准。排序依据依次为：架构不可逆性、是否改变 P0 数据契约、是否改变 P1 最小闭环、后续调整成本。

### 1. Q1（最高影响）：是否保留六角格战术层（已批准：2026-07-23）

**问题**

- **A．推荐：保留为可选微操层。** 郡域历史节点是必经战场；接战时可选自动、标准或六角微操。复用现有 S10 最大，工作量中等。
- B．六角成为每次接战必经层。战术最细，但节奏慢、UI 和 AI 工作量最大。
- C．彻底取消六角，只保留郡域节点 + 标准/自动。架构更轻，但会废弃大量现有能力。

本文按 A 展开。

**推荐方案**

推荐 **A：保留为可选微操层**。

**理由**

这最符合 0-A“先用小切片验证架构”的原则：P1 可直接复用已经存在的 S10 六角能力验证完整回写链，而无需同时证明每场战斗都适合微操。P2～P6 仍可独立扩展标准模式和自动结算，不会让单一表现层绑死郡域战场。

**影响范围**

- **影响 P0：是，间接影响。** P0 的 Encounter/模板契约必须预留局部战术地图入口，但无需实现六角玩法。
- **决策风险：关键、高改动成本。** 选 B 会把六角性能、AI、地图生产变成每场战争的硬门槛；选 C 会废弃既有 S10 投资。后期改选会重做状态流、UI 入口和验收标准。

### 2. Q8：分期路线（已批准：2026-07-23）

**问题**

批准的 P0～P6 顺序明确：

- P0 只做资料/Schema 契约；
- P1 采用“静态郡域 + 既有六角”的最小闭环；
- P2 把场景栈和中途存档列为正式门禁；
- P3 把战场 AI 行动选择也收口到权威 RNG，从“结算确定”提升为“整场复现”；
- P4 完成两个原型郡对照后，P5 才进入核心战线批量扩展。

**推荐方案**

推荐**整体批准 P0～P6 的既定顺序**：允许 P0 只做资料/Schema；接受 P1 最小闭环；将场景栈和中途存档列为 P2 门禁；P3 将战场 AI 行动选择纳入权威 RNG；完成 P4 两郡对照验收后再进入 P5 批量扩展。

**理由**

这条路线把资料正确性、玩法闭环、状态管理、动态随机和数据扩量依次解耦，正是 0-A 先小后大的工程哲学。尤其先完成两个地貌不同的原型郡，可在 P5 批量录入前暴露 Schema、工具和生成器缺陷，避免全量返工。

**影响范围**

- **影响 P0：是，直接决定能否启动以及 P0 的验收边界。**
- **决策风险：关键、高改动成本。** 若跳过 P0/P2 或过早进入 P5，错误会扩散到数据、存档和生成工具；具体阶段内部任务仍可后续调整，但阶段依赖顺序不宜频繁改变。

### 3. Q4：南郡采用哪个历史时间切片（已批准：2026-07-23）

**问题**

- A．东汉中期《郡国志》口径，适合建立基准模板。

- C．不区分年代，只做通用南郡；成本最低但历史错误会固化。

**推荐方案**

推荐 **B：以具体可玩剧本年份为准，以《郡国志》建立基线，再应用建安时期改置覆写**。

**理由**

P0 可以先验证“基线模板 + 年代覆写”的最小 Schema，而不必立即录入所有年代；这一结构能自然扩展到 P4～P6 的其他郡和剧本。若先做无年代的通用南郡，稳定 ID、县隶属和存档模板版本都可能在扩量时返工。

**影响范围**

- **影响 P0：是，直接影响 `validFromYear`、`validToYear`、`variantOf`、剧本覆写及模板版本契约。**
- **决策风险：关键、高改动成本。** 年代模型若后补，会影响静态数据、战场存档和迁移规则。
- **史料充分性：方向足够，定稿不足。** 已登记的[中国哲学书电子化计划《后汉书》](https://ctext.org/hou-han-shu/zh)与[维基文库《后汉书》卷一百一十二](https://zh.wikisource.org/zh/%E5%BE%8C%E6%BC%A2%E6%9B%B8/%E5%8D%B7112)足以支持《郡国志》基线，但两者主要是同一原典传统的不同入口，不足以单独证明建安改置。P0 定稿前还需以《三国志》及裴注等公有领域原典校勘具体剧本年份的析置、迁治和隶属变化。

推荐 **A：县城在战争实例内可占领、驻守并影响补给与结算，但不进入全局经济模拟**。

这能在 P1 真实验证“历史县城为何值得存在”，同时把经济、AI、迷雾和全局存档扩展留到 P5～P6。它比纯背景节点更能检验玩法，又避免违反 0-A 小数据集先验证核心链路的原则。

**影响范围**

- **影响 P0：是，P0 Schema 必须分离 `CountyDefinition`、战场节点状态与完整行政 `City`，但无需实现控制玩法。**
