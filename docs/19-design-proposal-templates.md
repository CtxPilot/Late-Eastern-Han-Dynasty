# 设计提案模板：地图、事件与外交

> 本文供 Issue/RFC 使用，帮助贡献者把想法写成可评审提案。**模板不是已定设计，示例字段不是运行时承诺。** 正式真源仍是 `docs/08-data-dictionary.md`、`docs/03-data-models.md`、`shared/types/` 与对应 Zod Schema。

## 一、通用提案头

```markdown
# [Sxx] 提案名称

- 状态：草案 / 待拍板 / 已批准 / 拒绝 / 被替代
- 作者与日期：
- 所属系统：S01~S22（一次只选一个主系统）
- 解决的问题：
- 当前代码/文档证据：
- 最小可验证切片：
- 明确不做：
- 数据真源是否变化：是/否；若是，列出 `08-data-dictionary.md` 变更
- 兼容与迁移影响：
- 验证路径与退出标准：
```

评审先问：它增加了什么玩家决策或框架能力？能否复用现有模型？是否创建第二套状态真源？若多种方案都合理，必须由维护者拍板后再改正式设计。

## 二、地图拓扑变更提案

当前运行时基线是 30 城、`shared/city-roads.ts` 官道邻接、`CampaignNode` 战役节点和 Natural Earth 底图。州是行政/检索维度，**不是当前可占领的单一节点**。任何“13州节点图”只能先作为替代方案 RFC，不能与城市图并存为第二套权威拓扑。

```markdown
## 地图拓扑变更

### 目标
- 解决的具体问题：导航、性能、战略抽象、剧本表现或其他
- 保留城市层：是/否
- 权威占领单位：城市 / 郡国 / 州 / 其他（只能有一个明确真源）

### 节点
| 稳定ID | 对应现有实体ID | 类型 | 控制权来源 | 显示层级 | 史地来源 |
|---|---:|---|---|---|---|

### 边
| 起点ID | 终点ID | 有向/无向 | 通行条件 | 规则影响 | 来源/设计理由 |
|---|---|---|---|---|---|

### 兼容审计
- `cities.json` / `city-roads.ts` / `CampaignNode` 如何迁移：
- 路径、迷雾、AI、行军、围城、事件条件受影响处：
- 旧剧本与存档兼容策略：
- 30城基线和0-B扩容的性能测量：

### 最小验证
- 地图可渲染和缩放
- 两个合法节点可寻路，非法边不可通行
- 一支 Army 可行军并正确进入围城/驻守状态
- 服务端控制权与客户端显示一致
```

禁止在提案阶段写死未经论证的州属性、兵力惩罚、季节倍率或势力分布；数值进入正式设计前须给出史料/平衡依据和测试方法。

## 三、历史事件卡（对齐当前 EventTemplate）

当前事件运行时不要求固定三个选项，也没有通用“延迟效果队列”或“历史偏离度”。需要这些能力时应另开 S14 数据模型 RFC，不得藏在单个事件 JSON 中。

```markdown
## 事件卡 [数字ID]

| 字段 | 内容 |
|---|---|
| name / category | historical / random / marriage / diplomacy / battle |
| sourceClass | official_history / annotated_history / literature / legend / gameplay |
| sources | 精确到篇目或卷次；玩法虚构写明 gameplay |
| scenarioIds | 允许出现的剧本ID |
| dateWindow | 起止年月；不是单一“触发年份” |
| decisionFactionId / decisionOfficerId | 谁决策；优先动态武将归属 |
| prerequisiteEventIds / mutexGroup | 前置与互斥 |

### 条件
| type | field | targetId | operator | value | 代码是否支持 |
|---|---|---:|---|---|---|

### 对话
| speakerId | speakerName | text | 来源层说明 |
|---:|---|---|---|

### 选项
| label | effects | AI基础权重 | 性格/理想修正 | 反事实边界 |
|---|---|---:|---|---|

### 失效与连锁
- 超出窗口如何失效：
- 前置选择不满足时是否永久阻断：
- 与哪些事件互斥：
- 角色、城市或势力消失时的 fallback：

### 验证断言
- 正确剧本/窗口/条件下触发
- 错误史料层、剧本或前置下不触发
- 玩家归属时进入 pending；AI归属时自动选择
- 每个效果均由当前 `EventEffect` 支持并命中正确目标
- 反事实选择能阻断不应发生的后续事件
```

事件进入 `events.json` 前，先更新共享类型/Zod（如确有新字段），再补 `verify-scenario-events.ts` 断言，最后实际走一遍玩家选择 UI。

## 四、外交机制扩展 RFC

当前 `DiplomacyLink` 代码真源只有双方势力、关系状态、`favorability` 与可选婚姻纽带。声望、信誉、衰减 modifier、外交点、贸易、联姻外交和动态联盟破裂等内容不能假装已经存在。

```markdown
## 外交扩展 RFC

### 玩家问题
- 当前外交为什么缺少有意义的决策：
- 可观测证据（试玩、日志、重复操作）：

### 最小机制
- 新增动作或状态：
- 与现有 `favorability/relation/marriageBond` 的关系：
- 是否需要新增持久化字段：
- 玩家能看到哪些信息，哪些属于情报：

### 状态转换
| 前置状态 | 命令 | 服务端校验 | 成功结果 | 失败结果 | 冷却/成本 |
|---|---|---|---|---|---|

### AI效用
- 输入必须来自当前可见/允许信息：
- 决策公式及每项权重的理由：
- 随机数如何注入以便复现：
- 禁止使用“某人物必然背叛”等不可解释硬编码：

### 跨系统影响
- S06迷雾、S07谍报、S09/S18人物资源、S14事件、S15 AI：
- API、UI、类型、Zod、存档兼容：

### 平衡与验收
- 至少三个代表局面（弱势求存/势均力敌/优势扩张）
- 防刷、防永久联盟、防无理由背叛规则
- 玩家可从 UI 和日志解释关系变化原因
- 自动模拟与实际点击验证路径
```

若提出“信誉”系统，必须先说明它与现有势力声望、友好度和总军师评价为何不能合并；只有独立决策价值成立时才新增字段。

## 五、不接受的提案写法

- 把建议性数值、伪代码或外部游戏惯例直接称为项目设计。
- 先造新 `EventManager`、EventBus 或客户端规则，再绕过现有服务端引擎。
- 同时重做地图、事件、外交三个大系统。
- 只写理想 UI，不列服务端权威状态、数据迁移和验证。
- 用“历史真实”“博弈论如此”代替史料、公式、试玩和可复现证据。
