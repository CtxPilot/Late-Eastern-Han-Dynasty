# 开发总则

## 一、文档驱动开发

- 任何新功能实施前，必须先在 `docs/` 中更新对应的设计文档（或确认已有设计足够落地）
- **每完成一个新功能，必须及时更新进度与相关文档**（强制，含 Demo/简化/占位实现）：
  1. `docs/10-progress.md` — 任务状态 + 会话日志（简化/占位须明确标注）
  2. 根目录 `HANDOFF.md` — 进度双写（阶段/怎么跑/约定/Next）
  3. **被该功能触及的全部相关文档** — 规则写在哪就改哪（常见：`04` 系统、`06` API、
     `07` UI、`03` 模型、`01` 摘要、`08` 数据字典等）
  - 禁止：只交代码不改文档；只改 `10-progress` 不改 HANDOFF；改了玩法却不改设计真源
- 设计决策变更必须同步更新 `02-architecture.md` 的决策记录
- **进度双写（强制）**：凡更新 `docs/10-progress.md`（勾选任务、写会话日志、改 0-A/0-B 状态摘要），
  **必须同步更新根目录 `HANDOFF.md`**，使下一次会话/下一个 agent 无需翻完整日志也能接手。
  - `10-progress.md`：任务勾选真源 + 完整会话日志  
  - `HANDOFF.md`：当前阶段、怎么跑、关键约定、已知简化、建议 Next（保持短、可扫读）  
  - 禁止只改其中一处；会话结束检查清单见 `11-context-management.md`

## 二、功能验证规则

每完成一个 task（对应 `10-progress.md` 中的一行），必须执行：

1. **类型检查**：`pnpm typecheck`
2. **Lint 检查**：`pnpm lint`
3. **API 测试**：如涉及 API，手动测试或编写 API 测试
4. **UI 验证**：如涉及 UI 组件，须在浏览器（或 headless）**实际点过**再报告完成，禁止仅凭代码/API 推断
5. **状态更新**：通过后，将 `10-progress.md` 中的 `[~]` 改为 `[x]`
6. **会话日志**：在 `10-progress.md` 底部"会话日志"中记录完成情况
7. **交接同步**：更新 `HANDOFF.md`（与第 5~6 步同次完成）

## 三、类型安全

- 所有静态数据（武将/城市/宝物/阵型/兵种）使用 TypeScript 类型 + Zod 运行时校验
- 禁止使用 `any`，仅在极少数必要情况使用 `unknown` + 类型守卫
- API 请求/响应必须有显式类型定义
- 类型文件集中在 `shared/types/` 目录

## 四、提交规范

- 每完成一个 Phase 的子集 task 后提交
- 提交信息格式：`[Phase-XX] 完成内容简述`
- 不提交 `/node_modules`、`/dist`、`/build` 目录
- 不提交任何 API 密钥、密码或环境变量文件

## 五、数据完整性

- 武将属性数据不可缺漏（1000+条必须完整，使用脚本生成骨架后人工校对）
- 城市坐标必须在东汉地图上准确定位
- 宝物效果必须自洽、不留未实现的 effect type
- 所有 JSON 静态数据文件提交前通过 JSON Schema 校验
- **核心数字真源规则**：武将/城市/宝物/技能等静态数据的规模数字，以 `08-data-dictionary.md`
  各文件条数说明为唯一真源；`01/09/10/11` 中的数字均为摘要引用。任何规模数字变更必须先改
  `08-data-dictionary.md`，再批量同步到其余文档，禁止只改单处。在逐条数据清单落地之前，
  条数视为设计目标；数据落地后须以实际 JSON 条数回写 `08-data-dictionary.md`。

- [x] 单挑系统

## 八、上下文管理

- 每次开发会话启动：先读根目录 `HANDOFF.md`，再读 `docs/10-progress.md` 定位 WIP
- 每次会话结束 / 每完成一个新功能：更新 `10-progress.md`（勾选 + 日志）**并同步** `HANDOFF.md`
  与相关设计文档（见第一节「完成即文档」）
- 不确定的设计点先问用户；有可玩性/架构收益的提议可主动提出
- 详见 `11-context-management.md`

## 九、禁止事项

- 禁止自创当代武将/女性角色
- 女性仅祝融可出战（其余为婚配/家族影响力/事件角色，**不可**像男将工作）
- 历史女角**不可**通过人事登用或美女资源系统获得（见 04§30.6；归 **家族 S18**）
- 子女必须为历史/演义/传说中有记载的人物
- 禁止引入不符合东汉末年背景的机制（如现代婚姻观、西方奇幻等）

## 十、命名分层规范

为避免不同层级的名称（如技能「天下无双」与势力特点「天下无双」）重复混淆，特制定以下规范：

> **全局原则**：跨层级名称不得相同；同层级名称必须唯一。新增名称前全库 `git grep` 查重。

| 层级 | 字数风格 | 示例 | 禁止冲突对象 |
|:----|:---------|------|:----------:|
| **技能/特性** (Skill/Trait) | 二字/三字为主 | 无双·武圣·龙胆·咆哮·天义·恶来·刚烈·虎痴·骑神·神将 | **战法名**、兵种名、势力特点名、计谋名 |
| **兵种/部队** (Unit) | 三字为主，史实称谓 | 陷阵营·虎豹骑·青州兵·白马义从·西凉铁骑·白耳兵·丹阳兵 | 技能名、势力特点名 |
| **势力特点** (Faction Trait) | **四字，出典可考** | 虓虎之勇·唯才是举·四世三公·坐断东南·仁德济世 | 技能名、兵种名、计谋名 |
| **战法/计策/计谋** (Ability/Plot) | 二字至四字 | 火计·美人计·釜底抽薪·调虎离山·暗渡陈仓 | 技能/特性名、势力特点名；**战法名推荐唯一，与兵种名相同时须改** |
| **阵型** (Formation) | 二字 | 方阵·圆阵·锥形·雁行·鹤翼·锋矢 | 无 |
| **宝物** (Item) | 史实/演义原器名 | 方天画戟·青龙偃月刀·赤兔马·孙子兵法 | 无 |

**全局约束**：
- 任何新命名，必须先 `git grep` 全库查询该名称是否已被其他层级占用
- 已占用时须换用同义但不同字面的名称，不得重名
- 四字词默认预留给「势力特点」层，其他层级新增命名原则上避免四字
