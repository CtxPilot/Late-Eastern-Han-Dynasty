### Gate 2：存档版本与随机性契约

存档建议使用独立信封，而不是直接在运行时 `GameState` 根部承担所有持久化职责：

```typescript
interface SaveEnvelopeV1 {
  schemaVersion: 1;
  createdAt: string;
  updatedAt: string;
  scenarioId: number;
  snapshot: GameStateV1;
}
```

- 加载路径必须是：解析 JSON → 按版本迁移 → 当前版本 Zod 校验 → 恢复运行时上下文。
- `isProcessing`、连接、网络重试、动画、选择框等瞬态状态不得进入存档。

### Gate 3：编排器与 Campaign 的职责审计

不预先决定采用 Command Bus、类式 Domain Service、Immer 或 StateMutation。先为 `game.ts` 每个导出操作建立调用图，再按以下信号拆分：

- 相同锁、权限、脱敏和广播样板重复出现；
- 一个函数同时做输入校验、随机解析、规则计算、跨系统结算和传输组装；
- 测试某条规则必须构造无关的全局运行时状态；
- 模块存在循环依赖或无法单独运行。

优先提取窄的命令处理函数和纯规则；权威状态提交仍由一个事务边界完成。事件/Mutation 日志只有在回放、审计、增量同步或跨系统订阅有明确消费者时才引入，避免为模式而模式。

Campaign 拆分候选：

1. `campaign-rules`：战力、补给、路径、围城概率等只读计算。
2. `campaign-resolution`：接受显式 RNG，产出结构化结果和 roll 记录。
3. `campaign-apply`：在单一事务内应用伤亡、归属、武将状态与战后结算。

是否返回完整不可变 `GameState`、局部 patch 或受控 draft，要通过性能与可测试性 spike 决定，不能在设计阶段硬选。

### Gate 4：0-B 性能基线

- 使用符合当前 Zod Schema 的合成数据或扩容夹具，不把不合法“假 GameState”作为结论依据。
- 至少测量：静态数据加载/校验、单回合结算、玩家状态裁剪+JSON序列化、关键路径规划、地图首次渲染/缩放/选择和客户端提交次数。
- 固定环境、数据规模、预热次数、重复次数与统计量（中位数、p95）；保存基线 JSON/Markdown，CI 先观察趋势，不立即用任意 50ms 阈值阻断。
- Profile 确认热点后再选择索引、memo、viewport culling、缓存或 Worker。105城不自动意味着需要空间索引；城市官道图与人物归属更可能适合 ID 索引/邻接表。

## 三、不直接采纳的建议

- **全套 Command Bus + StateMutation 一步到位**：尚无证据证明收益大于迁移成本。
- **所有系统通过事件总线互相通信**：会模糊服务端权威调用链，且事件顺序/事务失败更难推理。
- **整份 GameState 快照测试作为主要集成策略**：快照过大、随机字段和无关排序会制造脆弱更新；优先断言领域不变量和关键结果，必要时对规范化子集做快照。
- **运行时不再 Zod 校验**：外部 JSON/存档仍是不可信边界；是否优化应先基准测量。
- **Canvas 动画一律用 ref 绕过 React**：只适用于高频表现性状态；权威位置和可交互终态仍须与声明式状态收敛，避免双真源。
- **宿主字体 fallback**：违反 FontBarrier 与工程字体资产闭环，明确禁止。

## 四、UI 与规则可见度

复杂系统需要解释性 UI，但不应把所有公式裸露给玩家。每个新规则至少提供：动作成本、关键正负修正、结算结果、失败原因和可追踪战报。百分比预测只有在数据确实可见且算法可稳定解释时展示；受迷雾影响的内容应给区间或“不明”，不能泄露服务端隐藏状态。

程序化头像继续使用参数化 SVG/CSS 与 `avatarGene`；`feTurbulence` 已在当前四人切片中使用。卷轴/印章动画属于 P5 体验增强，必须遵守减少动态效果偏好和性能预算，不能先于完整头像数据层。
