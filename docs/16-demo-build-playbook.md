# 0-A 特色展示版可玩流程

> 场景：`英雄集结·开局即高光（0-A Demo）`
> 推荐势力：刘备军（需在 ScenarioSelect 选择）
> 建议长度：12 个战略回合；六角战斗内部回合不计入战略回合
> 本文用 `[已验证可玩]`、`[引擎已验证/入口受阻]`、`[替代展示]`、`[设计中/待实现]` 区分真实能力，禁止把替代体验或仅引擎可跑写成正式可玩机制。

## 一、范围结论

### 已验证可玩的链路

| 系统 | 实际状态 | Demo 入口 |
|------|----------|-----------|
| 回合/内政 | `[已验证可玩]` | 结束回合；农/商/城开发、征兵、训练、施米 |
| 战役编成 | `[已验证可玩]` | 主将 + 副将 + 智力≥85参谋 + Squad 五部；行军、围城、强攻 |
| 六角战斗 | `[已验证可玩]` | 道路邻接出征；移动、攻击、火计、战法、暴击/反击/连击 |
| 单挑 | `[引擎已验证/入口受阻]` | `verify-duel.ts` 全断言通过；实际六角流程可移动到相邻并创建单挑状态，但接受挑战后 API 因嵌套请求锁返回400，当前不能标为完整可玩 |
| 谍报/计谋 | `[已验证可玩]` | 本流程使用“招募密探→探秘番禺”；四种 L1 计谋已有引擎，但 UI 目标城需先取得 `detailed` 情报 |
| 外交 | `[已验证可玩]` | 进贡、献美、结盟；本流程使用刘备—孙权结盟 |
| 人事/家族 | `[已验证可玩]` | 搜索、登用、三轨任命、婚配、妻随夫、子女登场 |

### 只作替代展示的节点

- `[替代展示] 史载部曲武将编成出征`：`officers.json` 已覆盖既定12名史载部曲持有者——许褚、曹仁、李典、吕虔、高顺、孙策、甘宁、徐盛、周泰、公孙瓒、张嶷、夏侯惇。当前只能把这些武将作为普通主将/副将编入 Army；没有 `personalTroops`、部曲随将迁移、损耗、补充或继承逻辑。臧霸作为第12个占位替换位补入0-A阵容，但不改变设计文档中既定12人硬编码名单。
- `[替代展示] 农业开发`：点击“开发农业”只执行现有 farm 开发，不是民屯田或军屯田。英雄集结的 `eventIds=[]`，不会触发历史事件。

### 设计中或待实现

- `[设计中/待实现] 部曲机制`：仅有规则设计，无共享类型、Zod、运行时状态、API或UI。
- `[设计中/待实现] 军屯/民屯机制`：仅有规则设计；不得用普通农业开发冒充屯田。
- `[设计中/待实现] 爵位编成加成`：场景能保存 `nobilityRank`，但当前副将上限代码只识别君主身份与武官官职，没有读取爵位值。
- `[设计中/待实现] 战役编成与六角单挑联动`：战役层的自动战斗含简化“单挑事件”，六角层另有完整单挑引擎；两者不是同一状态链。真正串联依赖 S21，本文分开展示。
- `[设计中/待实现] S17 L2/L3`：本Demo只使用已实装的 L1 四计，不把11种战略计谋和8种国策写成可玩能力。
- `[设计中/待修复] 单挑 UI/API 接受链路`：`battleChallengeDuel()` 已持有 `withLock` 时又调用同样加锁的 `battleDuelStep()`，接受挑战必然命中“操作处理中”错误。该缺陷属于运行时代码，Session 104 依照“仅数据整合”边界只记录、不修复。
- `[条件可玩] L1计谋 UI`：计谋面板以目标城为入口并要求 `detailed` 情报；开局宛、番禺等目标均禁用。服务 API 可直接发起离间不代表开局 UI 已满足条件，因此不列为必经节点。

## 二、开局阵容与演示靶点

场景是 `whatif` 且启用 `noLifespan`，用于跨年代集结，不宣称历史同年共事。

| 地点 | 阵容/归属 | 演示目的 |
|------|-----------|----------|
| 江陵（刘备） | 刘备、诸葛亮、张飞、赵云、公孙瓒 | 主将/副将/参谋即时编成；公孙瓒作为史载部曲武将替代展示 |
| 襄阳（刘备） | 关羽 | 六角战斗固定攻方武将 |
| 宛（曹操） | 许褚 | 六角战斗固定守方武将与单挑对手 |

## 三、12回合推荐流程

### 开局准备（190年1月，结束第1回合前）

1. `[已验证可玩] 阵容巡览`：启动 `pnpm --filter @leh/shared build && pnpm dev`，页面会自动创建场景1与刘备军。展开左侧“己方城池”，查看江陵、襄阳；再查看地图上的宛、建业、下邳，确认上述名将已经就位。
2. `[已验证可玩] 人事任命`：选中江陵，展开“人事 → 任命”，选择赵云，切到“武官”，任命为“将军”。这一步展示三轨官职与能力门槛；它不是主副将槽位操作。
3. `[已验证可玩] 外交触点`：展开“外交”，在孙权军卡片点击“结盟”。开局友好40，满足≥30门槛；消耗500金后关系变为“同盟”，并获得盟友情报共享。
4. `[已验证可玩] 谍报动作`：保持江陵为当前城，展开“谍报”，点击“在江陵招募密探”；在“派遣任务”选择新密探、`探秘`、目标“番禺”，点击“派出（探秘）”。任务成功或失败均完成本节点；失败时密探进入恢复期也属正常。若探秘获得 `detailed` 情报，可再到“计谋”面板发起离间/假情报；这不是必经条件。
5. `[替代展示] 农业开发`：选中江陵，展开右侧“内政操作”，点击一次“开发农业”。预期消耗100金并提高农业；这只是现有 farm 开发，用来替代展示“重视屯田/生产”的主题，不是屯田机制。

### 第1回合：战役编成与史载部曲武将替代展示

6. `[已验证可玩] 主副将+参谋编成`：保持选中江陵，展开左侧“战役”。选择：
   - 主将：刘备
   - 副将：张飞、赵云、公孙瓒
   - 参谋：诸葛亮（智力100，满足智≥85）
   - 目标：番禺
   - 推荐兵种/阵型：重骑/锋矢
   - 兵力：4000；粮草：1500
7. 点击“编成出征”。Army 中应显示主将、副将、参谋与 Squad 阵位。`[替代展示]` 公孙瓒在此只是普通副将；“白马义从随将出征”没有独立兵数、经验或随将状态。
8. 点击“结束回合”。进入190年2月后，刘备军应抵达官道相邻的番禺并进入围城。检查谍报任务记录、密探恢复状态与行动日志。

### 第2回合：战役层结果

9. `[已验证可玩]` 在“战役”中选中刘备军，可先尝试参谋行动或直接“强攻”。打开自动战斗报告，观察主副将、参谋、阵型、兵种与士气/组织度共同进入战力计算。结果含随机性；若失败，以报告和撤退链路为展示结果，不要求重开刷胜。
10. 结束第2回合，观察驻守恢复或撤退状态，以及行动日志中的补给/战果变化。

### 第3—5回合：独立六角战斗与单挑引擎触点

11. `[已验证可玩]` 回到大地图后选中敌城“宛”，展开右侧“军事操作”，点击“出征攻城”。道路只允许从襄阳出发，六角战斗攻方会选关羽、守方会选许褚。
12. 点击“选择我军”，沿蓝色可移动格逐回合接近许褚；每次移动或行动后用“结束行动（敌军自动出手）”推进战术回合。不要先耗尽关羽气力。
13. `[引擎已验证/入口受阻]` 双方相邻且关羽气力≥20时，点击“单挑”，再点许褚所在六角格。当前请求会在创建内部单挑状态后因嵌套锁返回400，不能把它演示成已跑通的面板流程。需要证明7指令与三向克制引擎时，另在 `server/` 执行 `pnpm exec tsx src/scripts/verify-duel.ts`；这只是引擎冒烟，不是 S21 串联或 UI 可玩验收。
14. 可继续用普通攻击、战法或火计观察暴击/反击/连击；随后完成战斗或点击“撤军返回”。这条六角战斗与第1回合创建的战役 Army 没有编成/状态联动，依赖 S21 的串联仍未实现。

### 第6—12回合：自由观察与收束

15. 第6—8回合：任选己方城执行农/商/城开发、征兵、训练或施米，并查看经济、人口与体力变化。密探恢复后可再次探秘；若某敌城达到 `detailed`，可选做离间或假情报，但不把随机情报深度与计谋成功列为验收条件。
16. 第9—10回合：展开“家族”查看已有婚配与妻随夫数据；也可手动释放合适武将后观察跟随检定，但不作为必经项。
17. 第11—12回合：复查“行动日志”“谍报记录”“外交关系”和战役 Army 状态，形成收束：同一局已跑通任命、外交、谍报、内政、战役编成、围城和六角战斗，并触达单挑入口与家族数据；L1计谋视情报条件选做，单挑完整结算只完成引擎级验证。

## 四、事件隔离说明

英雄集结是关闭历史事件的假想剧本，`eventIds=[]`；190事件只在《关东义兵》场景扫描。屯田机制仍未实装，也不再由旧“屯田令”样例事件替代展示。

## 五、验证口径

- `pnpm validate-data`：验证10类静态数据、0-A数量、13州覆盖和城市投影，不验证UI流程。
- `generate-0a-data.ts` 已因结构过时冻结为 fail-fast；更新生成器前禁止重跑，以免覆盖手工维护的兵种、30史实武将和两剧本数据。
- `pnpm test`：当前只覆盖 shared 层3个测试文件、68个纯函数测试，不是全仓或服务端引擎测试。
- 服务端引擎需另跑：`verify-duel.ts`、`verify-crit.ts`、`verify-campaign.ts`、`verify-fire-tactic.ts`、`verify-child-engine.ts`、`pnpm verify-scenario-events`。
- 本文中“已验证可玩”只指已有运行时代码且实际流程成功；引擎冒烟通过但 UI/API 受阻必须单列。“替代展示”与“设计中/待实现”均不得用于宣称正式功能完成。
