# 学习与贡献路线：以本仓库为实践场

> 面向希望一边学习前端、一边参与历史策略模拟框架的开发者。本文件不是速成就业承诺，也不改变 `docs/09-roadmap.md` 的工程优先级。

## 一、先建立最小 Web 基础

学习目标不是背 API，而是能够阅读并修改一个小组件：

1. HTML 语义、表单、无障碍名称与键盘交互。
2. CSS 盒模型、Flex/Grid、响应式布局、层叠与字体加载。
3. JavaScript 模块、数组/对象、事件、Promise 与 `async/await`。
4. Git 的分支、diff、commit、rebase/merge 基础和 Pull Request 流程。

仓库练习：为一个只读面板修复布局或可访问性问题，并通过 `pnpm typecheck` 与浏览器实际点击验证。

## 二、进入本项目技术栈

| 领域 | 在本仓库中的落点 | 应掌握的边界 |
|---|---|---|
| TypeScript | `shared/types/`、前后端函数签名 | 静态类型不能替代 JSON 运行时校验 |
| React | `client/src/components/` | 组件声明 UI，不直接修改权威游戏状态 |
| Zustand | `client/src/stores/gameStore.ts` | 保存服务端状态投影与 UI 状态，不复制规则引擎 |
| Konva/Canvas | 地图、战斗、程序化头像 | 场景坐标、屏幕坐标、缩放、命中与重绘成本 |
| HTTP/WebSocket | `client/src/services/api.ts`、服务端 routes/ws | REST 负责命令与状态，WS 负责推送；都不能绕过服务端校验 |
| Express | `server/src/routes/`、`services/` | 路由、编排、纯规则分层 |
| Zod/JSON | `shared/validators/`、`server/src/data/` | 先 schema、后数据；规模数字服从数据字典 |
| Vitest/验证脚本 | `shared/*.test.ts`、`server/src/scripts/verify-*.ts` | 静态检查不是玩法验证，功能还需走真实操作路径 |

推荐练习顺序：读一个共享纯函数及测试 → 修一个 React 只读组件 → 跟踪一次 API 命令全链路 → 为服务端纯规则补测试 → 最后才新增跨层功能。

## 三、Web 游戏专项知识

- 游戏状态与表现分离：规则输出数据，渲染读取数据；不要把 Konva/DOM 对象写进 `GameState`。
- 坐标系统：区分历史经纬度、地图逻辑坐标、Canvas 世界坐标与屏幕坐标。
- 性能测量：先用浏览器 Performance/React Profiler 找瓶颈，再考虑缓存、视口裁剪、对象复用或 Worker；不要预先套用优化清单。
- 确定性：随机结算应可注入或记录 seed/roll，便于复现战报和跨实现测试。
- 可访问性：复杂 Canvas 画面之外仍需 HTML 操作入口、文本状态与键盘路径。
- 资产管线：尺寸、色域、字体、来源、许可、哈希与生成记录都属于工程数据。

## 四、策略游戏设计与非编程知识

### 游戏设计

学习核心循环、反馈、机会成本、信息不完全、决策密度、随机性控制、失败恢复和教程节奏。每次提出新系统先回答：它增加了什么新决策，是否与已有系统交叉，能否用更小规则验证。

### 历史研究

至少理解东汉州郡地理、官职与爵制、军事组织、人口与粮运、人物关系和关键时间线。区分正史、注疏、后世演义与本作虚构层，并记录可追溯出处。不要把短视频或二手解说作为唯一真源。

### UX 与信息设计

大战略界面要管理信息层级、比较成本和决策疲劳。优先让玩家看清“当前状态—可执行动作—代价—结果”，再添加装饰。使用真实可玩路径做可用性测试，而不是只看静态截图。

### 视觉、声音与版权

学习构图、对比、重复、对齐、留白、色彩与字体可读性。音画素材必须保留许可和来源证据；文物本体、网站数字文件与生成式输出的权利状态不可混为一谈，详见 `docs/00-dev-constitution.md` §11。

### 概率、经济与 AI

掌握基础概率分布、期望值、方差、机会成本、边际收益与简单效用函数。AI 首先要可解释、可复现、遵守同一规则，再追求“聪明”。平衡结论必须来自模拟或试玩数据。

### 开源维护与发布

学习 Issue 复现、代码审查、语义化版本、变更日志、安全披露、许可证兼容和社区沟通。能让陌生贡献者从干净 checkout 复现问题，比单次写出更多代码更重要。

## 五、关于未来迁移 Godot 或其他引擎

策略模拟的数据和规则语义通常比渲染更容易迁移，但不存在脱离代码审计的固定“复用百分比”。本仓库可以保留 JSON 合约、规则说明和测试向量；React/Tailwind/Konva/Zustand 相关表现与交互通常需要重写，TypeScript 引擎若跨语言也需要移植和重新验证。

决定迁移前，只做一个有验收条件的技术 spike：读取场景 JSON、渲染少量节点、推进一回合、执行一个确定性战斗夹具。记录代码量、工时、性能、调试体验和发布目标，再决定保持 Web 客户端、采用新客户端连接现有服务端，还是重写本地引擎。

## 六、学习产出而不是学习清单

每个阶段都应形成仓库可验证产物：一个测试、一个小 PR、一份来源审校、一段性能基线或一次可用性记录。避免同时学习三套框架；当前贡献默认围绕 React + TypeScript + Konva，Godot 仅在明确迁移决策后进入正式路线。
